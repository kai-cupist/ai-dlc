# 통합 테스트 지침

## 목적
MSW mock → 실제 backend API 전환 후, 프론트엔드와 백엔드가 올바르게 통신하는지 검증합니다.

## 사전 준비

### 1. 전체 스택 시작
```bash
# 1. DB 시작
docker-compose up -d db

# 2. 백엔드 시작
cd backend
uv run alembic upgrade head
uv run python scripts/seed.py
uv run uvicorn app.main:app --port 8000 &

# 3. 프론트엔드 실제 API 모드로 시작
# customer-web (.env에 VITE_API_MODE=real, VITE_API_URL=http://localhost:8000/api)
cd ../apps/customer
npm run dev &

# admin-web
cd ../admin
npm run dev &
```

## 통합 테스트 시나리오

### Scenario 1: 고객 주문 플로우 (customer-web ↔ backend-api)
| 단계 | 액션 | 기대 결과 |
|------|------|----------|
| 1 | 고객앱 접속 | 메뉴 화면 표시 (시드 데이터 메뉴) |
| 2 | 카테고리 사이드바 클릭 | 해당 카테고리로 스크롤 |
| 3 | 메뉴 선택 (옵션 있는 메뉴) | 옵션 선택 화면 표시 |
| 4 | 옵션 선택 + 장바구니 추가 | 장바구니에 메뉴+옵션 추가 |
| 5 | 장바구니 → 주문하기 | 추천 바텀시트 표시 |
| 6 | 바로 주문하기 | 주문 확정 화면 (계산 안내 멘트) |
| 7 | 주문 확정 | 주문번호 표시, 대기 화면 이동 |
| 8 | 대기 화면 | 진행 상황 표시 (모래시계) |
| 9 | 영수증 확인 | 회차별 영수증 표시 |
| 10 | 주문 내역 | 주문 목록 + total 표시 |

### Scenario 2: 관리자 모니터링 플로우 (admin-web ↔ backend-api)
| 단계 | 액션 | 기대 결과 |
|------|------|----------|
| 1 | 관리자 로그인 | 대시보드 표시 |
| 2 | 대시보드 확인 | 테이블별 카드, 2초 폴링 |
| 3 | 주문 카드 클릭 | 주문 상세 모달 |
| 4 | 상태 변경 (대기중→준비중) | 상태 업데이트 반영 |
| 5 | 상태 변경 (준비중→완료) | 완료 상태 반영 |
| 6 | 이용 완료 | 테이블 리셋, 주문 아카이빙 |
| 7 | 과거 내역 확인 | 아카이빙된 주문 표시 |

### Scenario 3: 관리자 메뉴 관리 (admin-web ↔ backend-api)
| 단계 | 액션 | 기대 결과 |
|------|------|----------|
| 1 | 메뉴 관리 페이지 | 카테고리별 메뉴 목록 |
| 2 | 메뉴 추가 | 새 메뉴 저장, 목록 갱신 |
| 3 | 메뉴 수정 | 변경사항 반영 |
| 4 | 옵션 그룹 생성 | 옵션 그룹 + 항목 저장 |
| 5 | 메뉴에 옵션 연결 | 고객앱에서 옵션 표시 확인 |
| 6 | 메뉴 순서 변경 (드래그) | 순서 저장, 고객앱 반영 |

### Scenario 4: 크로스 앱 연동
| 단계 | 액션 | 기대 결과 |
|------|------|----------|
| 1 | 고객앱에서 주문 생성 | 관리자 대시보드에 2초 내 표시 |
| 2 | 관리자가 상태 변경 | 고객앱 대기 화면에 반영 |
| 3 | 관리자가 주문 삭제 | 고객앱 주문 내역에서 제거 |
| 4 | 관리자가 이용 완료 | 고객앱 주문 내역 초기화 |

## 정리

# 모든 프로세스 종료

docker-compose down
